# 09. Поверхность атак

## Общая характеристика

Проект `python_auth` является локальным desktop-приложением с GUI. Основная поверхность атак связана с пользовательским вводом, который передаётся через поля логина и пароля.

Сетевой интерфейс, REST API и парсинг пользовательских файлов в текущей версии отсутствуют.

## Точки входа

| Точка входа | Описание | Связь с fuzzing |
|---|---|---|
| Поле логина | Пользователь вводит имя учётной записи | Генерация и мутация строк |
| Поле пароля | Пользователь вводит пароль | Генерация и мутация строк |
| Кнопка входа | Запускает проверку логина и пароля | Проверка реакции на разные входы |
| Функция хеширования | Преобразует пароль в SHA-256 | Проверка строк разной длины и кодировки |
| Функция аутентификации | Сравнивает введённые данные с базой пользователей | Проверка логики обработки входов |
| Словарь пользователей | Используется для поиска логина | Проверка неизвестных и специальных логинов |
| Счётчик попыток | Ограничивает число попыток входа | Проверка логики состояния |

## Пользовательский ввод

Основная поверхность атак — это поля ввода логина и пароля.

Потенциально опасные входы:

- пустые строки;
- очень длинные строки;
- Unicode-строки;
- управляющие символы;
- строки с нулевыми байтами;
- строки с кавычками;
- строки, похожие на SQL-инъекции;
- строки, похожие на shell-инъекции;
- пути к системным файлам;
- визуально похожие Unicode-символы.

## Бизнес-логика аутентификации

Для fuzzing наиболее релевантна не сама GUI-форма, а внутренняя логика:

- вычисление SHA-256-хеша;
- поиск пользователя по логину;
- сравнение хеша введённого пароля с эталонным значением;
- возврат результата аутентификации;
- обработка некорректных строк.

Именно эта часть тестируется инструментами Hypothesis, HypoFuzz и Radamsa.

## Связь точек входа с инструментами

| Компонент | Hypothesis | HypoFuzz | Radamsa |
|---|---|---|---|
| Логин | Генерация строк | Coverage-guided запуск property-based теста | Мутация seed-входов |
| Пароль | Генерация строк | Coverage-guided запуск property-based теста | Мутация seed-входов |
| `myhash` | Проверка строк и Unicode | Проверка покрытия | Косвенная проверка через аутентификацию |
| `authenticate` | Проверка результата и отсутствия исключений | Проверка покрытия и устойчивости | Проверка мутированных входов |
| GUI | Не тестируется напрямую | Не тестируется напрямую | Не тестируется напрямую |

## Парсинг файлов

В текущей версии приложение не принимает пользовательские файлы и не выполняет парсинг внешних файлов.

Следовательно, fuzzing файловых форматов к текущей версии не применяется.

Если в будущем будет добавлена загрузка базы пользователей из файла, импорт конфигурации или работа с SQLite-дампами, эта область должна быть включена в fuzzing.

## API и сетевые интерфейсы

В текущей версии приложение не предоставляет:

- REST API;
- GraphQL API;
- TCP/UDP-сервер;
- HTTP-интерфейс;
- сетевой протокол.

Следовательно, инструменты для сетевого fuzzing, такие как `boofuzz`, имеют ограниченную применимость.

## Уязвимые компоненты

### 1. Хеширование пароля

Пароль хешируется алгоритмом SHA-256.

Потенциальные проблемы:

- отсутствие соли;
- отсутствие специализированного алгоритма хранения паролей;
- одинаковые пароли дают одинаковые хеши.

### 2. Глобальная база пользователей

Пользователи и хеши паролей хранятся в словаре.

Потенциальные проблемы:

- база пользователей зашита в коде;
- отсутствует централизованное управление пользователями;
- нет отдельного защищённого хранилища.

### 3. Обработка логина

Потенциальные проблемы:

- отсутствие ограничения длины;
- отсутствие нормализации Unicode;
- возможные ошибки при специальных символах;
- визуально похожие Unicode-символы могут восприниматься пользователем как одинаковые, но обрабатываться программой по-разному.

### 4. Обработка пароля

Потенциальные проблемы:

- отсутствие ограничения длины;
- отсутствие политики сложности;
- отсутствие соли;
- возможное повышенное потребление ресурсов на больших строках.

### 5. GUI-слой

Потенциальные проблемы:

- зависимость от графического окружения;
- сложность автоматизированного тестирования;
- смешение GUI и бизнес-логики.

GUI-слой не является основной целью fuzzing в текущей работе, так как его тестирование требует отдельной методики GUI automation.

## Связь поверхности атак с fuzzing

Fuzzing позволяет проверить:

- устойчивость к некорректным логинам;
- устойчивость к некорректным паролям;
- устойчивость функции хеширования;
- корректность обработки Unicode;
- отсутствие аварийных завершений;
- отсутствие обхода аутентификации;
- корректность обработки граничных значений.

## Ограничения текущей поверхности атак

Текущая версия проекта имеет ограниченную поверхность fuzzing, так как отсутствуют:

- сетевые интерфейсы;
- API;
- парсинг файлов;
- сложные структуры данных;
- внешняя база данных;
- десериализация.

Поэтому основное тестирование сосредоточено на обработке строкового пользовательского ввода.

## Вывод

Проект подходит для fuzzing. Наиболее релевантная поверхность атак — обработка пользовательского ввода в логине и пароле.

API, сетевые интерфейсы и парсинг файлов в текущей версии отсутствуют, поэтому не являются основными объектами fuzzing.

Для текущей версии проекта выбранная связка Hypothesis, HypoFuzz и Radamsa является достаточной, так как она покрывает генерацию входов, coverage-guided выполнение и мутацию seed-корпуса.
