# 05. Обоснование выбора инструментов

## Выбранные инструменты

Для практической части выбраны три инструмента:

1. Hypothesis.
2. HypoFuzz.
3. Radamsa.

Выбор сделан так, чтобы покрыть три разных подхода к fuzzing:

- generation-based fuzzing;
- coverage-guided fuzzing;
- mutation-based fuzzing.

## Hypothesis

`Hypothesis` выбран как основной инструмент генерации входных данных.

Причины выбора:

- поддерживает Python;
- легко интегрируется с `pytest`;
- позволяет задавать стратегии генерации строк;
- хорошо подходит для проверки логина и пароля;
- умеет минимизировать найденные failing examples;
- позволяет собирать покрытие через `pytest-cov`.

Применение в проекте:

- fuzzing функции `myhash`;
- fuzzing функции `authenticate`;
- проверка обработки строковых и бинарных входов.

## HypoFuzz

`HypoFuzz` выбран как coverage-guided инструмент для практической части.

Причины выбора:

- поддерживает Python;
- работает поверх property-based тестов;
- интегрируется с pytest-based тестированием;
- использует информацию о покрытии кода;
- не требует libFuzzer, AFL или отдельной нативной сборки;
- подходит для macOS-среды, где возникли сложности с установкой Atheris;
- хорошо подходит для небольшого Python-проекта с бизнес-логикой обработки строк.

Применение в проекте:

- запуск property-based тестов для `authenticate`;
- запуск property-based тестов для `myhash`;
- сбор покрытия по модулю `src/auth_logic.py`;
- проверка устойчивости к пустым, длинным и Unicode-строкам;
- фиксация результатов через pytest-логи и coverage-отчёты.

Ограничения:

- `HypoFuzz` не создаёт отдельные mutation-input файлы;
- его эффект фиксируется через pytest/coverage, а не через отдельную директорию собственных отчётов;
- в небольшом проекте с малым количеством ветвлений прирост покрытия может быть ограниченным.

## Radamsa

`Radamsa` выбран как mutation-based fuzzer.

Причины выбора:

- не зависит от языка программирования;
- прост в запуске;
- хорошо подходит для мутации seed-корпуса;
- позволяет моделировать искажённые пользовательские входы.

Применение в проекте:

- мутация seed-файлов формата `login:password`;
- передача мутированных входов в Python harness;
- сохранение crash-входов при падениях.

## Почему не выбраны остальные инструменты

### Atheris

`Atheris` был рассмотрен как coverage-guided fuzzer для Python, однако не выбран для практической части.

Причины отказа:

- при установке в используемой среде возникли проблемы с отсутствием libFuzzer;
- на macOS стандартный Apple Clang не содержит libFuzzer;
- инструмент требует более сложной нативной инфраструктуры сборки;
- для текущего учебного проекта использование `Atheris` оказалось избыточным.

Вместо него выбран `HypoFuzz`, так как он лучше подходит для текущей среды и интегрируется с уже используемыми Hypothesis-тестами.

### PythonFuzz

`PythonFuzz` применим к проекту, но не выбран для основной практической части, потому что его роль пересекается с `HypoFuzz`.

Дополнительно были учтены возможные сложности совместимости с современными версиями Python и зависимостями.

### boofuzz

`boofuzz` не выбран, так как в текущей версии приложения отсутствует сетевой интерфейс.

### Schemathesis

`Schemathesis` не выбран, так как проект не предоставляет REST API, GraphQL API или OpenAPI-схему.

## Итоговое обоснование

Выбранные инструменты покрывают разные техники fuzzing:

| Инструмент | Подход |
|---|---|
| Hypothesis | Generation-based |
| HypoFuzz | Coverage-guided |
| Radamsa | Mutation-based |

Такой набор является достаточным для проверки текущей версии приложения, так как основная поверхность атак связана с пользовательским вводом логина и пароля.
