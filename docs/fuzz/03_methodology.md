# 03. Методика фаззинг-тестирования

## Общий подход

Для проекта `python_auth` используется комбинированная методика fuzzing-тестирования:

1. Generation-based fuzzing.
2. Coverage-guided fuzzing.
3. Mutation-based fuzzing.

Такой подход выбран потому, что приложение написано на Python и основная поверхность атак связана с пользовательским вводом логина и пароля.

## Обоснование подхода

GUI-слой `tkinter` напрямую не является удобным объектом fuzzing, так как он зависит от графического окружения. Поэтому тестирование проводится на уровне бизнес-логики аутентификации.

Для этого целесообразно выделить функции:

- `myhash(value)`;
- `authenticate(login, password)`;
- обработку строки формата `login:password`.

Такой подход позволяет тестировать приложение автоматически, без запуска GUI.

## Seed-корпус

Начальный корпус входных данных включает валидные и невалидные пары логин/пароль.

Примеры seed-входов:

```text
admin:@dm1n
vovuas:2003
admin:wrong
unknown:password
:
админ:пароль
用户:密码
admin:пароль🔐
' OR '1'='1:test
$(whoami):password
../../etc/passwd:test
```

## Способы генерации данных

### 1. Generation-based fuzzing

Используется инструмент `Hypothesis`.

Он генерирует:

- случайные строки;
- пустые строки;
- длинные строки;
- Unicode-строки;
- строки со специальными символами;
- бинарные данные, преобразованные в строку.

### 2. Coverage-guided fuzzing

Используется инструмент `HypoFuzz`.

`HypoFuzz` применяется как coverage-guided слой поверх property-based тестов на базе `Hypothesis` и `pytest`. В отличие от внешних fuzzing-движков, он не выступает отдельным генератором входных файлов, а работает внутри pytest-based пайплайна и использует информацию о покрытии кода для более эффективного исследования пространства входных данных.

В рамках проекта `HypoFuzz` применяется для проверки функций:

- `myhash(value)`;
- `authenticate(login, password)`;
- обработки строковых входов;
- обработки Unicode-входов;
- обработки пустых и длинных строк.

Так как приложение имеет небольшую бизнес-логику и ограниченное количество ветвлений, основной результат использования `HypoFuzz` фиксируется через pytest-логи и coverage-отчёты, а не через отдельные crash-артефакты.

### 3. Mutation-based fuzzing

Используется инструмент `Radamsa`.

Он мутирует заранее подготовленные seed-файлы и передаёт результат в тестовый harness. Этот подход позволяет проверять устойчивость к искажённым реальным входным данным.

## Критерии успешности

Fuzzing считается успешным, если:

- тестируемая функция не падает;
- не возникает необработанных исключений;
- не возникает зависаний;
- результат аутентификации всегда имеет тип `bool`;
- неправильные данные не приводят к успешной аутентификации;
- crash-входы при наличии сохраняются в артефактах;
- покрытие кода зафиксировано в отчётах.

## Фиксация ошибок

Для каждой найденной ошибки фиксируются:

- инструмент, который обнаружил ошибку;
- входные данные;
- stack trace;
- команда запуска;
- дата и время запуска;
- критичность;
- путь к crash-файлу;
- рекомендации по исправлению.

## Длительность тестирования

Минимальная длительность тестирования:

| Инструмент | Длительность / количество |
|---|---|
| Hypothesis | 3000 примеров на тест |
| HypoFuzz | pytest-based запуск с фиксацией покрытия |
| Radamsa | 1000 мутированных входов |

## Влияние обратной связи

`HypoFuzz` используется как coverage-guided слой поверх Hypothesis/pytest и ориентируется на покрытие кода при исследовании входных данных.

`Hypothesis` не является классическим coverage-guided fuzzer, но умеет генерировать большое число входных данных и минимизировать найденные failing examples.

`Radamsa` не использует обратную связь по покрытию и работает как black-box mutation-based fuzzer.

## Обработка зависаний и падений

При обнаружении зависаний или падений:

1. Входные данные сохраняются в `docs/fuzz/appendix/crashes`.
2. Лог ошибки сохраняется в `docs/fuzz/appendix/logs`.
3. Ошибка классифицируется по критичности.
4. Формируется задача разработчику.
5. После исправления crash-вход добавляется в регрессионные тесты.

## Особенности HypoFuzz в текущем проекте

В текущем проекте `HypoFuzz` не создаёт отдельные crash-файлы и собственные HTML-отчёты. Для фиксации результатов используется запуск через `pytest` с подключением `pytest-cov`.

Результаты сохраняются в виде:

- логов запуска;
- coverage-отчётов;
- итоговой информации о прохождении тестов.

Это связано с тем, что `python_auth` имеет небольшую и детерминированную бизнес-логику, а основная тестируемая поверхность ограничена обработкой строк логина и пароля.
